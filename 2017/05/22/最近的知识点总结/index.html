关键词

最近加入新公司，感觉需要学的知识太多了，不过，最要还是搞安卓应用，framework层这一块。所以还算是比较熟悉的领域，我上周研究了一阵子MVP模式。

MVP模式

这里我们不得不提到MVC模式，MVVM模式。

MVC模式

在前2年，软件开发架构讨论最多的还是MVC模式，即，模型-视图-控制器模式。这里借一张图来说明：

其中，Model层用来获取业务数据，关于数据的CRUD，都可以放到这里完成；
View层则负责展示数据，刷新界面数据；
Control会负责执行业务逻辑;

3者关系一般是单向控制的。即：
control -> model -> view -> control

但并不一定，有时候，我们可以在view层控制，发送命令给control，继续后面的控制；
有时候，我们可以在control层发出命令，改变model，继续后面的控制；

MVP模式 - MVC的改进版

对于MVC模式，model和view可以直接交互，但我们有时候不希望这样，希望所有的界面展示都由一个中间代理presenter来控制，而界面变化后，也由presenter通知model进行数据更改。这里，因此就有了这样的关系：

model <-> presenter <->view

和MVC的区别在于，model和view不再直接交互。二者的交互都通过p进行。

MVVM模式

对于有些应用，可能界面和数据高度耦合，很难区分开，那么我们就会把presenter层改成ModelView层，
3者的关系就变成这样：

Model <-> ModelView ->View
即Model和ModelView是双向通信的，二者变化互相影响，当ModelView变化后，就会引起View的变化。
